# 単語出題フォーマットの選別（バリエーション削減）

## 目的・背景

復習クイズは、1単語につき **27形式 × 各3変種（`VARIANTS_PER_FORMAT = 3`）= 最大81問** をサーバ側で生成・保存している。
素材（定義・例文・類義語・活用形・イラスト等）が揃った語では実測で **20〜24形式 × 3 ≈ 60〜72問/語** が生成される。

出題時（`ReviewSessionPlanner.pick`）は、その語の保存問題から比率調整して **1問だけ** を選んで表示するため、大半は「重複回避のための予備」でしかない。一方で：

- 形式が多すぎて **役割が重複**（意味想起3方向、綴り3系統、例文リスニング5系統、活用形3系統…）
- **AI 生成コスト**（1語あたり非イラスト ≈ 17〜19形式 × 3 ≈ 50件超を Claude で生成）と **保存量** が形式数に比例
- 教育的価値の低い形式（例文全文ディクテーション、聞き分け等）まで一律に生成

→ 形式を **バランスよく選別**し、モダリティ（テキスト／音声／イラスト、4択／入力）の網羅は保ちつつ総数を削減する。

## 現状調査（数の実態）

- 定数: `VARIANTS_PER_FORMAT = 3`（`backend/src/quizQuestions.ts:10`）
- 形式総数: **27**（AI生成23 + イラスト規則生成4）
- 単一の真実源:
  - backend `AI_FORMAT_SPECS`（`quizQuestions.ts:116-405`）+ `generateIllustrationQuestions`（`quizQuestions.ts:593-651`）
  - iOS `enum ReviewQuestionFormat`（`ios/.../Support/FormatSelector.swift:6-68`。rawValue が DB `quiz_questions.format` と一致）
- 出題可否は各形式の `isAvailable(info)` に依存。**常時利用可**は `tc9 / vc2 / vc5 / vt1` の4形式のみ。他は定義・例文・類義語・対義語・コロケーション・活用形・品詞・イラストの有無で開閉。
- 保存: `quiz_questions` テーブル、`UNIQUE(word, target_language, format, variant_index)`（`db.ts:267-282`）。iOS では生成せずサーバ取得のみ。

## 選別方針

判断基準（この順で残す/削る）:

1. **役割の重複を排除** — 同一スキル×同一モダリティが複数ある場合は最良の1つに集約
2. **モダリティ網羅は維持** — 出題（テキスト／音声／イラスト）× 回答（4択／入力／イラスト4択）の各枠に最低1形式は残す
3. **教育的価値の低い形式を削る** — 全文ディクテーション(vt2)、聞き分け(vc6)、類似音(vc5)、品詞(tc8)、文中語義(tc10) 等
4. **常時利用可の形式は最低限に** — tc9/vc2/vc5/vt1 のうち綴り系が重複 → vc2・vt1 に集約

## 選別（確定：Tier B）

**採用は Tier B（27→15形式、変種 `3→2`）。** 1単語あたり最大 **15 × 2 = 30問**。
「絵を選ぶ」回答形式（`tc11`/`vc8`）は廃止し、イラストは *絵→単語*（`ic1`/`it1`）出題のみ。リスニングは重要形式を厚く残す（🎧6形式）。

- **残す(15):**
  - テキスト出題: `tc2`(単語→定義) / `tt1`(定義→入力) / `tc3`(例文穴埋め) / `tc4`(類義語) / `tc5`(対義語) / `tc6`(コロケ) / `tc7`(活用形)
  - 音声出題🎧: `vc1`(音声→定義) / `vc3`(定義音声→単語) / `vc2`(音声→綴り) / `vc4`(例文→単語聞き取り) / `vtt1`(例文リスニング穴埋め入力) / `vt1`(単語ディクテ)
  - イラスト出題: `ic1`(絵→単語4択) / `it1`(絵→単語入力)
- **削除(12):** `tc1` `tc8` `tc9` `tc10` `tc11` `tt3` `vc5` `vc6` `vc7` `vc8` `vtc1` `vt2`
- **集約・削除の根拠:**
  - イラストの「絵を選ぶ」回答（`tc11`/`vc8`）を廃止 → イラストは *絵→単語* の `ic1`/`it1` のみ
  - 意味想起「定義→単語」は `tt1`(入力) に集約し `tc1`(テキスト4択) を削除
  - 綴りは `vc2`(音声4択) + `vt1`(入力) に集約し `tc9`(テキスト綴り) を削除
  - リスニング穴埋めは入力版 `vtt1` を採用し、4択版 `vtc1` を削除
  - 活用形は `tc7`(4択) に集約し `tt3`(入力)/`vc7`(リスニング) を削除
  - 低価値/冗長として `tc8`(品詞)/`tc10`(文中語義)/`vc5`(類似音)/`vc6`(例文聞き分け)/`vt2`(例文全文入力=高負荷) を削除
  - リスニングは `vc1`/`vc2`/`vc3`/`vc4`/`vtt1`/`vt1` の6形式を確保（単語の意味/綴り・定義→単語・文中の語聞き取り・文脈穴埋め・ディクテーション）

## 影響範囲

- **backend**
  - `quizQuestions.ts`: `AI_FORMAT_SPECS` から該当エントリ削除／`generateIllustrationQuestions` のイラスト4形式を調整／必要なら `VARIANTS_PER_FORMAT` を変更
  - `db.ts`: 削除形式の旧データ purge を追加（tt2 と同様の `DELETE FROM quiz_questions WHERE format IN (...)` を起動時マイグレーションに追記。`db.ts:287` が前例）
- **iOS**
  - `FormatSelector.swift`: `ReviewQuestionFormat` から削除ケースを除去、`promptBucket`/`answerBucket` の switch を更新。（未知形式はデコード時に自然除外されるため後方互換は保たれるが、enum は掃除する）
  - `FormatRatioTargets.v1`（`FormatSelector.swift:89-92`）: 削減後の形式分布に合わせて目標比率を **再チューニング**（例: 音声形式を減らした場合、音声50%目標は過大になり得る）
- **再生成**: 既存単語の問題は削除形式ぶんが purge され、次回生成/再生成時に新セットへ。全単語一括再生成の要否を判断（`POST /api/quiz-questions/generate`、管理画面の再生成ボタン）

## テスト方針

- backend: `generateQuizQuestions` が残存形式のみ返すことを確認（型・件数）。purge SQL の冪等性確認
- iOS: `ReviewQuestionFormat` のケース削除後もビルド通過、`FormatSelector.select` が残存形式でバランス選定できること。旧形式データを含む取得レスポンスがデコードで自然除外されること
- 管理画面 `quiz_questions` 表示で1語あたり件数が想定内に収まることを目視確認
- 実機で復習セッションを一巡し、モダリティが偏らず出題されること

## 実装メモ（Phase 2-3）

- 廃止した「絵を選ぶ」回答（illustrationChoices）と一致率判定（matchRateThreshold）は、現行フォーマットでは未生成になったが iOS のモデル/レンダリング（`ReviewSessionView.illustrationChoiceGrid`・`ReviewAnswerJudge` の一致率）は将来の再利用に備え残置（dead code だが無害）。
- 旧データの扱い: 削除12形式は起動時 purge で消える。残存15形式の既存行は variant 0..2 のまま残り、次回の生成/再生成で variant 0..1 に作り直される（変種2に減っても出題は無害）。
