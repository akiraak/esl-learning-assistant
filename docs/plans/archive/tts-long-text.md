# サーバ側TTSの長文対応調査・対応

## 目的・背景

- `/api/tts` は現在 `TTS_TEXT_MAX_LENGTH = 2000` 文字で入力を拒否している
- レッスン本文の読み上げ（一般再生プレイヤー化のタスク）を見据えると、2000文字を超える長文を音声化できるか確認が必要
- Gemini 2.5 TTS (`gemini-2.5-flash-preview-tts` / `gemini-2.5-pro-preview-tts`) は非ストリーミングの `generateContent` で音声を inline base64 で返す構成のため、長文時に「入力制限」「出力音声の打ち切り」「タイムアウト」のいずれかで破綻する可能性がある

## 対応方針

### Phase 1: 実測調査

- `.env` の `GEMINI_API_KEY` を使い、検証スクリプト（scratchpad）で文字数を段階的に増やして `synthesizeSpeech` 相当の呼び出しを実施
  - 目安: 約1,000 / 2,000 / 4,000 / 8,000 文字
- 各ケースで以下を記録
  - HTTPステータス / エラー内容
  - レイテンシ
  - 生成されたPCMのバイト数 → 音声秒数（24kHz mono 16bit）
  - **音声が末尾まで読まれているか（打ち切りの有無）** — 文字数に対する音声秒数の比率で判断
- 結果をこのファイルに追記する

### Phase 2: 結果に応じた対応

- そのまま長文が通る場合: `TTS_TEXT_MAX_LENGTH` を実測上限に合わせて引き上げるのみ
- 上限がある/打ち切りが起きる場合: サーバ側で文単位にチャンク分割し、各チャンクを順に合成してPCMを連結 → 1つのWAVとして返す
  - キャッシュ（`tts_audio` テーブル）は最終的な結合WAV単位（既存の textHash キー）で従来どおり機能させる

## 影響範囲

- `backend/src/tts.ts`（分割合成ロジック）
- `backend/src/index.ts`（`TTS_TEXT_MAX_LENGTH`）
- iOS / 管理画面は API 互換のまま（リクエスト/レスポンス形式は変更しない）

## テスト方針

- 検証スクリプトで長文（8,000文字程度）を合成し、WAVの秒数が文字数に見合うこと・再生して末尾まで読まれることを確認
- 既存の短文リクエストがキャッシュ含め従来どおり動くことを確認

## 調査結果（2026-07-02 実測, gemini-2.5-flash-preview-tts / Leda）

| 文字数 | 結果 | レイテンシ | 音声長 | 読み上げ速度 |
|---|---|---|---|---|
| 986 | 1回目: finishReason=OTHER・音声なし / 2回目: 成功 | 109s / 23s | 76.0s | 13.0 chars/s |
| 1,973 | 1回目: 成功 / 2回目: HTTP 500 INTERNAL | 67s / 24s | 199.6s | 9.9 chars/s |
| 3,950 | 成功（ただし**サイレント打ち切り**: STOPなのに音声が期待の6割） | 71s | 195.9s | 20.2 chars/s |
| 7,910 | 成功（打ち切りなし） | 228s | 655.1s | 12.1 chars/s |

### 結論

- **長文（8,000文字）でも生成自体は可能**。ハードな入力上限には当たらなかった
- ただし以下の理由で1リクエスト一括合成は実用に耐えない
  1. `finishReason=OTHER`（音声なし）や HTTP 500 が散発し、現行実装にはリトライがない
  2. `finishReason=STOP` でも音声が途中で切れる**サイレント打ち切り**が発生する（4,000文字で観測）
  3. 長文ほどレイテンシが増大（8,000文字で約4分）し、失敗時のやり直しコストが大きい

### 対応（Phase 2 で実装）

- 文境界で最大約1,500文字のチャンクに分割し、チャンクごとに合成 → PCM連結 → 1つのWAVで返す
- チャンクごとにリトライ（HTTPエラー / 音声なし / finishReason≠STOP / 音声長が極端に短い場合）
- チャンク単位のリクエストにタイムアウトを設定
- `/api/tts` の `TTS_TEXT_MAX_LENGTH` を 2,000 → 20,000 に引き上げ（API互換は維持、キャッシュは従来どおり全文ハッシュ単位）

## Phase 2 実装結果（2026-07-02）

- `backend/src/tts.ts`
  - `splitTextIntoChunks()`: 文境界（`.!?！？。…` + 改行）で最大1,500文字に分割。1文が上限超えなら単語境界、空白なしの超長トークンは文字数でハード分割
  - `synthesizeChunk()`: チャンク単位で最大3回リトライ。リトライ条件は HTTPエラー（4xxは429以外即時失敗）/ 音声なし / `finishReason≠STOP` / 読み上げ速度が30 chars/s超（サイレント打ち切り検知）。1リクエスト120秒タイムアウト
  - `synthesizeSpeech()`: チャンクを同時3並列で合成し、元の順序でPCM連結 → 1つのWAVに
- `backend/src/index.ts`: `TTS_TEXT_MAX_LENGTH` 2,000 → 20,000
- 検証スクリプト: `backend/scripts/tts-long-check.ts`（分割ユニットチェック + 実API合成）

### 検証結果

- 分割ロジック: 6〜4,999文字・空白なし3,200文字・超長1文で全チャンクが上限以内、文字落ちなし
- 4,000文字の実合成（以前サイレント打ち切りが起きた長さ）:
  - 逐次版: 309.8→292.4s の正常な音声（13.5 chars/s、打ち切りなし）だが所要 約16分
  - 並列版: 309.8s の正常な音声（12.7 chars/s、打ち切りなし）、所要 約2.5分
  - 生成WAVは `afinfo` で 24kHz mono Int16 として正常と確認
