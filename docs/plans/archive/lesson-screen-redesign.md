# Lesson画面の作り直し — レイアウト案

## 目的・背景

TODO「Lesson画面の作り直し」に対応するレイアウト検討資料。要件は以下の3点。

1. クラスとレッスンの作成と選択ができる
2. 選択中のクラスとレッスンがわかりやすく表示される
3. クラス（のレッスン）の中にコンテンツ（画像・文字起こし・翻訳）、単語、問題がある

現行実装の課題:

- 選択中クラスはツールバー左上のボタンだが、ラージタイトル表示中はシェブロンだけになり
  クラス名がほぼ見えない。レッスン名もリストのセクションヘッダーで目立たない
- クラス/レッスンの切り替え・作成がシート（`ClassLessonSwitcherView`）の奥にあり、
  選択状態と切り替え操作が画面上で結びついていない
- コンテンツ・単語・問題が1本の縦リストに積まれ、レッスンが育つと写真一覧に押されて
  単語・問題が画面外に沈む

補足（データモデルとの関係）: [data-model.md](../specs/data-model.md) では
コンテンツ（Photo）・単語（WordOccurrence）・問題（Question）は **Lesson に属する**。
「クラスの中に〜がある」という要件は「クラス⊃レッスン⊃コンテンツ」の意図と解釈するが、
クラス単位の集約ビュー（クラス内全レッスン分をまとめて見る）を望む場合は別途確認が必要
（→ 案ごとに影響を記載）。

## 案A: ヘッダーカード型（現行改良）

現行の縦リスト構成は維持し、「選択中の表示」を大きなカードにして解決する最小変更案。

```
レッスン
┌─────────────────────────────┐
│ ESL Beginner A                    │ ← クラス名（小・グレー）
│ Unit 3 Reading                 ▾ │ ← レッスン名（大・太字）
└─────────────────────────────┘
   タップで切り替えシート（クラス/レッスン作成もここから・現行流用）

[📷 写真を追加]

── コンテンツ (3) ──────────────
[thumb] 10/12  完了
[thumb] 10/11  失敗
── 単語 (12)  ────────────────
apple   りんご                    >
── 問題 (5)   ────────────────
（今後実装）
```

- 作成/選択: 現行の切り替えシートを流用。カード全体がタップ領域なので導線が明確
- 選択中表示: カードで常時表示。スクロールしてもヘッダーに固定（pinned）にできる
- コンテンツ: 現行と同じ縦一列セクション
- 長所: 実装コスト最小。現行UIテストもほぼ流用可能
- 短所: 「写真が多いと単語・問題が沈む」問題は残る。切り替えは相変わらずシート1枚奥

## 案B: チップ切り替え型（選択と表示を一体化）

クラスとレッスンを横スクロールのチップ2段で常時表示し、タップで即切り替える案。
コンテンツ領域はセグメントで「コンテンツ/単語/問題」を切り替える。

```
レッスン
クラス:   [ESL Beginner A ✓] [ESL B] [＋]
レッスン: [Unit 3 ✓] [Unit 2] [Unit 1] [＋]

[ コンテンツ | 単語 | 問題 ]   ← セグメント
┌─────────────────────────────┐
│ [📷 写真を追加]                   │
│ [thumb] 10/12  完了               │
│ [thumb] 10/11  失敗               │
└─────────────────────────────┘
```

- 作成/選択: チップタップで即切り替え（シート不要）。末尾の[＋]でその場で名前入力して作成
- 選択中表示: 選択チップのハイライトで一目瞭然。2段なので階層（クラス→レッスン）も見える
- コンテンツ: セグメント切り替えで各領域が画面全体を使える（写真が多くても単語が沈まない）
- 長所: 切り替え最速（1タップ）・選択状態が最も分かりやすい・毎日の4ステップと相性が良い
- 短所: クラス/レッスンが増えると横スクロールが長くなる（個人利用なら実用上問題は小さい）。
  実装コストは中（チップUI・セグメント切り替えの新規実装）

## 案C: ドリルダウン型（クラス一覧→レッスン一覧→詳細）

iOS標準の階層ナビゲーションに寄せる案。起動時は前回開いていたレッスン詳細を復元する。

```
[クラス]              [ESL Beginner A]        [Unit 3 Reading]
ESL Beginner A (3) >   Unit 3 Reading (3) >    [コンテンツ|単語|問題]
ESL Beginner B (1) >   Unit 2 Vocabulary  >    [thumb] ...
[＋ クラスを追加]       [＋ レッスンを追加]
```

- 作成/選択: 各階層の一覧＋追加ボタン。管理操作としては最も自然
- 選択中表示: 「今いる場所＝選択中」。ナビバーの戻るボタンで階層も分かる
- コンテンツ: 詳細画面はセグメントまたはセクション
- 長所: クラス/レッスン数が増えてもスケールする。iOS標準作法で迷わない
- 短所: 日常フロー（毎日同じレッスンに写真を足す）で毎回2階層潜るのはタップ数が増える。
  screen-design.md v1で「ドリルダウンしない」方針にした経緯と逆行（復元で緩和はできる）

## 案D: ダッシュボード型（サマリカード＋ハブ）

選択ヘッダーの下に「コンテンツ/単語/問題」のサマリカードを並べ、タップで各一覧へ飛ぶハブ構成。

```
レッスン
ESL Beginner A ▾ ・ Unit 3 Reading ▾   ← それぞれ独立メニューで切替・作成
─────────────────────────────
[📷 写真を追加]   [📝 単語を追加]
┌─────────┐ ┌─────────┐ ┌─────────┐
│コンテンツ │ │  単語    │ │  問題    │
│   3件    │ │  12語   │ │   5問   │
│ [thumb]  │ │ apple…  │ │ 正答80% │
└─────────┘ └─────────┘ └─────────┘
     ↓タップで各一覧（フル画面）へ
```

- 作成/選択: ヘッダーのクラス名/レッスン名それぞれがメニュー（Menu）になっており、
  切り替えと「＋新規作成」を提供
- 選択中表示: ヘッダーに常時表示（テキストメニューなので場所を取らない）
- コンテンツ: サマリ＋ドリルイン。一覧は専用画面なので件数が増えても破綻しない
- 長所: screen-design.md v1の「2.1 ホーム」に最も近く、既存設計資産を活かせる。
  クラス集約表示（クラス全体の単語数など）をカードに足しやすい
- 短所: 目的の情報まで常に1タップ挟まる。カードUI・各一覧画面の実装コストは中〜大

## 比較

| | 切り替えの速さ | 選択状態の見やすさ | 件数増への耐性 | 実装コスト |
|---|---|---|---|---|
| 案A ヘッダーカード | △（シート経由） | ○ | △（縦積みのまま） | 小 |
| 案B チップ | ◎（1タップ） | ◎ | ○（セグメントで分離） | 中 |
| 案C ドリルダウン | △（毎回2階層） | ○（=現在地） | ◎ | 中 |
| 案D ダッシュボード | ○（メニュー） | ○ | ◎ | 中〜大 |

## 推奨

**案B（チップ切り替え型）** を推奨する。要件2「選択中がわかりやすい」への回答が最も直接的で、
クラス/レッスンの作成・選択（要件1）も画面から離れずに完結する。想定ユーザー（個人・
受講クラス数は少数）ではチップの横スクロール増加も実用上問題になりにくい。
写真・単語・問題の表示領域をセグメントで分けることで、現行の「単語・問題が沈む」課題も解消する。

## 決定事項

- 採用案: **案A（ヘッダーカード型）**。ただし**クラスとレッスンの作成は別の画面で行う**
  （切り替えシート内のアラート入力をやめ、シートから遷移する専用フォーム画面にする）
- コンテンツの表示単位: レッスン単位（推奨どおり。ユーザーからの指定は無かったため、
  クラス集約が必要になったら別タスクで対応）

## 実装ステップ

1. `LessonsView`: ツールバーのクラスボタンを廃止し、リスト先頭に「クラス名＋レッスン名 ▾」の
   ヘッダーカードを常設（タップで切り替えシート）。「新しいレッスン」ボタン＋アラートを削除。
   「写真」セクションを「コンテンツ」に改名
2. `ClassLessonSwitcherView`: 選択専用にし、作成はプッシュ遷移に変更
   - 「クラスを追加」→ `ClassAddView`（新規フォーム画面。作成後にそのクラスを選択して戻る）
   - クラスヘッダーの「＋」→ `LessonAddView`（新規フォーム画面。作成後にそのレッスンを
     選択してシートごと閉じる）
3. UIテスト（`testClassLessonCaptureFlow` / `testWordAddFlow`）をフォーム画面経由の
   作成フローに追随
4. `screen-design.md` 2.1/2.2章を更新
5. xcodegen再生成 → ビルド → シミュレータでUIテスト実行

## 影響範囲

- iOS: `LessonsView` / `ClassLessonSwitcherView` / 新規 `ClassAddView` `LessonAddView` / UIテスト
- docs: `screen-design.md` 2.1/2.2章

## テスト方針

- シミュレータでのUIテスト（クラス/レッスン作成→選択→コンテンツ/単語/問題表示）を採用案に合わせて更新
