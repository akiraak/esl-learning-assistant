# 複数ユーザーで使う場合の実装検討（調査資料）

> ステータス: **調査のみ（実装なし）**。方針を決めるための材料整理。
> 最終確認したい未決事項は末尾「§8 確認したいこと」にまとめた。

## §1. 目的・背景

現状このアプリは「作者が自分の英語学習に使う」前提で、学習データは 1 端末のローカルに閉じている。
「複数ユーザーで使う」を実装検討するにあたり、まず **「複数ユーザー」が何を指すのか** を分解し、
現状アーキテクチャのどこが単一ユーザー前提になっているかを棚卸しし、取りうる設計オプションと
影響範囲を整理する。結論の実装は行わず、方針決定の判断材料を提示する。

## §2. 現状アーキテクチャの整理

### 2.1 iOS（クライアント）— 学習データはすべて端末ローカル

- 学習データは **SwiftData でローカルにのみ** 保存。同期・バックアップ・アカウントの概念は無い。
  - モデル: `Class → Lesson → { Word / WordOccurrence / Photo / AudioClip / Composition }`
  - `ESLLearningAssistantApp.swift` で `ModelContainer(for: Class, Lesson, Photo, Word, WordOccurrence, Composition, AudioClip)` を素の（=ローカルのみ）構成で生成。
  - **データは 1 端末に固定**され、機種変更・iPad 併用・再インストールで失われる／持ち運べない。
- 端末ごとの設定は `UserDefaults`（`AppSettingsKeys`）:
  - `backendBaseURL` / `apiSecret` … バックエンド接続先と共有シークレット
  - `targetLanguageCode`（母語。既定 `ja`）… AI 生成の説明言語。**ユーザー個人設定だが端末ローカル**
  - `ttsModel`
- 生成アセット（単語情報・イラスト・TTS・クイズ）は都度バックエンドから取得しキャッシュ。

### 2.2 バックエンド（Node/Express + SQLite、`https://esl.chobi.me`）

- 役割は実質 **AI 生成プロキシ + 生成物の共有キャッシュ**。ユーザーの学習データは持たない。
- 認証は **単一の共有シークレット** `API_SECRET`（`X-API-Secret` ヘッダ、timing-safe 比較）。
  - `index.ts` で `/api/*` に一律適用。`/admin` はエッジ（Cloudflare Access）、`/health` は無認証。
  - **ユーザー識別子は一切存在しない**（`user` テーブルも `user_id` カラムも無い）。
- SQLite は全テーブルが **コンテンツアドレス / グローバル共有キャッシュ**:
  - `words`（`UNIQUE(word, target_language)`）, `word_illustrations`（`key_hash`）,
    `tts_audio`（`text_hash`）, `quiz_questions`（`UNIQUE(word, target_language, format, variant_index)`）
  - つまり **生成物は言語スコープで全ユーザー共有**。これは意図的な割り切りで、複数ユーザー時はむしろ
    「一度誰かが生成した単語情報は全員で使い回せる」利点になる。
  - 個人化の余地（`/api/word-info` の `context` = 教科書ページ本文、`userTranslation`）はあるが、
    保存キーは `(word, language)` のみで **last-write-wins**（`generation_count` 加算）。個人別には保存されない。
- ログ・料金（`requests` / `word_info_requests` / `writing_feedback_requests` / `system_logs` / `pricing_state`）
  も **グローバル**。ユーザー別のコスト按分・上限管理は無い。

### 2.3 一言でいうと

> **「学習データ = 端末ローカル・単一ユーザー」「バックエンド = 個人データを持たない共有 AI キャッシュ」** という構造。
> 複数ユーザー対応の主戦場は **iOS 側の学習データ（同期／分離／持ち運び）** であり、バックエンドの
> 共有キャッシュ部分は多くのシナリオでそのまま活かせる。

## §3. 「複数ユーザー」の 4 つの意味（シナリオ分解）

要件がどれかで難易度と実装方針が大きく変わる。**まずここを決めるのが最優先。**

| | シナリオ | 例 | 主な必要要件 | 相対難易度 |
|---|---|---|---|---|
| **A** | 同一人物・複数端末 | iPhone + iPad、機種変更 | 学習データの同期 or 移行 | 低〜中 |
| **B** | 複数人・各自独立 | 家族数人がそれぞれ自分の学習に使う | データのユーザー別分離 | 中 |
| **C** | 複数人・データ共有/協働 | 先生が複数生徒を管理、共有教材 | 共有データモデル + 権限 | 高 |
| **D** | 不特定多数へ配布 | App Store 公開 | アカウント + コスト按分/上限 + 濫用対策 | 非常に高 |

- **A と B は同じ仕組みでほぼ同時に解ける**（後述の CloudKit プライベート DB は「iCloud アカウント単位で
  自動分離 + 端末間同期」なので、A の同期と B の分離を一挙に満たす）。
- **C** はデータ共有・権限の設計が加わり別物。現時点では対象外と割り切るのが妥当か要確認。
- **D** は「個人ツール」から「サービス」への質的ジャンプ。共有 API キーで全ユーザーの有料 AI 呼び出しを
  肩代わりする現状のままでは **コスト青天井 + 濫用リスク**。専用の深掘りが必要（§6）。

## §4. 単一ユーザー前提の棚卸し（何が壊れる / 足りないか）

1. **学習データがローカル専用・識別子なし** → 端末をまたげない・分離できない・共有できない。（最重要）
2. **共有 API シークレット 1 個** → ユーザー別の認証・失効・レート制限・コスト按分ができない。
   シークレットを知る全員がフルアクセス。1 人だけ止める／課金を分けることが不可能。
3. **バックエンドキャッシュは `(word, language)` グローバル** → 共有は利点だが、`context` による
   個人別の語義最適化は保存されない（last-write-wins）。B/C で「人によって違う語義説明」が要るなら要設計。
4. **母語設定が端末ローカル**（`targetLanguageCode`） → アカウント化するならユーザー属性に移すべき。
5. **コスト・ログがグローバル** → D では per-user メータリング／上限が必須。
6. **管理画面（/admin）が全データ横断** → マルチテナントなら閲覧範囲の設計が要る。
7. **生成物ファイルが無制限に蓄積**（`data/illustrations`, `data/tts`） → 配布規模では容量・寿命管理が要る。

## §5. 設計オプション

学習データ層と認証層は **別々の軸** として検討できる。

### 5.1 学習データ層（本丸）

#### オプション 1: CloudKit プライベート DB 同期（推奨候補）

- SwiftData の CloudKit 連携（`ModelConfiguration(cloudKitDatabase: .automatic)` 相当）で、
  学習データを **ユーザーの iCloud プライベート DB** に置く。
- **Apple が肩代わりしてくれるもの**: 本人認証（iCloud アカウント）、ユーザー別の完全分離、
  端末間の自動同期・衝突解決の下地、バックアップ。→ **A（複数端末）と B（複数人分離）を同時に解決**。
- **バックエンド改修は原則ゼロ**（バックエンドは個人データを持たない AI キャッシュのままでよい）。
- **コスト**: Apple Developer Program + iCloud/CloudKit 前提。**Apple 専用**になる（Web/Android 不可）。
- **⚠ SwiftData モデルの CloudKit 要件（要改修・後述 §5.3）**:
  - すべてのプロパティは **オプショナル or デフォルト値必須**、
  - すべてのリレーションは **オプショナル必須**、
  - **`@Attribute(.unique)` 使用不可**。
- **C（共有）** は CloudKit 共有 DB（`CKShare`）で一応可能だが複雑。まずは対象外が現実的。

#### オプション 2: バックエンド・アカウント + 同期 API

- `user` テーブル + 認証（Sign in with Apple 等で発行するユーザートークン/JWT）を導入し、
  学習データをバックエンドへ複製、同期エンドポイントを実装。
- **利点**: フルコントロール。将来の Web/Android、C（共有）、D（配布）まで拡張可能。
- **欠点（大）**: 同期・衝突解決の自前実装、SwiftData を「サーバ状態のキャッシュ」に作り替え、
  サーバ側ストレージ/バックアップ、そして **個人学習データをサーバで預かる責任（プライバシー・法対応）**。
  バックエンドが「個人データを持たないプロキシ」から「ステートフルな個人データ基盤」へ質転換する。

#### オプション 3: 手動エクスポート / インポート（低コストの安全網）

- SwiftData を JSON 等にエクスポート → 別端末でインポート。ライブ同期・アカウント無し。
- **A の「機種変更でデータを失わない」だけ**を安価に満たす。同時併用や自動分離は不可。
- Phase 0 の保険として単独でも価値がある（同期実装前でもデータ消失を防げる）。

#### オプション 4: ハイブリッド（推奨の落としどころ）

- **個人学習データ = CloudKit プライベート DB**（オプション 1）で A/B を解決、
- **バックエンド = 従来どおり共有 AI キャッシュ**のまま（改修最小）。
- 生成物の共有という現状の利点を保ちつつ、個人データだけを iCloud で分離・同期する。

### 5.2 認証層（バックアクセス）— 学習データ層とは独立に選べる

- **少人数の信頼グループ（家族数人 = B）**: 現状の共有シークレットのままで十分。
  グローバルキャッシュはむしろ利点（生成コスト共有・高速化）。**追加実装ほぼ不要。**
- **不特定多数（D）**: per-user 資格情報 + レート制限 + **コスト上限** + Sign in with Apple +
  濫用対策が必須。共有キーのままでは 1 ユーザーの乱用が全体課金に直結する。→ §6。

### 5.3 CloudKit 採用時に必要な iOS モデル改修（具体）

現状モデルを確認した結果、**致命的な非互換は無いが、リレーションの optional 化が必要**:

- ✅ `@Attribute(.unique)` は **どのモデルにも無い**（`grep` 済み）。CloudKit と相性良好。
  - 参考: 過去の「クラス内でレッスン名ユニーク」(`unique-lesson-title-per-class`) は
    DB 制約ではなくアプリロジックで担保している想定 → CloudKit でも維持可能（要確認）。
- ⚠ **非オプショナルの to-one リレーションが CloudKit 非対応**:
  - 例: `Lesson.schoolClass: Class`（非オプショナル）、`WordOccurrence` の `word` / `lesson` など。
  - CloudKit は全リレーションを optional 要求 → `Class?` 等へ変更 + 参照箇所のアンラップ対応が要る。
- ⚠ **スカラーはデフォルト値 or optional 必須**: `id`/`title`/`createdAt` 等は init でのみ既定を与えており、
  プロパティ宣言にデフォルトが無い。CloudKit 用に宣言側デフォルト付与 or optional 化の見直しが要る。
- 既存ローカルストアからの移行は、過去に何度も踏んだ **SwiftData 軽量マイグレーション地雷**
  （非オプショナル追加でストアが開けなくなる、埋め込み Codable の CodingKeys 問題）に再び注意。
  `WordReviewState` の nullable ストレージ + computed 既定の手法が参考になる。

## §6. シナリオ D（公開配布）固有の論点 ※採用する場合は別途詳細検討

- **コスト**: 全ユーザーの OCR/翻訳/単語情報/イラスト/TTS/作文添削は **有料 API**。共有キーのままだと
  青天井。per-user メータリング、無料枠 + 上限、または BYO API キー方式の検討が必要。
- **認証・濫用対策**: Sign in with Apple、レート制限、不正利用検知。
- **プライバシー/規約**: 個人データ保管（オプション 2 採用時）、App Store 審査、プライバシーポリシー。
- **管理画面**: マルチテナント化 or 管理者専用スコープ。
- → これは「個人ツールの複数ユーザー化」ではなく「プロダクト化」。**本調査の範囲外**として、
  必要になった時点で専用スペックを起こすのを推奨。

## §7. 推奨アプローチ（要件確定後の想定ロードマップ）

**大前提: §8 で目的（A/B/C/D のどれか）を確定してから着手する。**

想定が「自分の複数端末 + 家族数人まで（A + B、Apple 専用で可）」なら、以下の段階案を推奨:

- **Phase 0（安全網・独立して有効）**: 学習データの手動エクスポート/インポート（オプション 3）。
  同期実装前でも機種変更でのデータ消失を防げる。小さく確実。
- **Phase 1（本命）**: CloudKit プライベート DB 同期（オプション 1/4）。
  - Step 1: モデルの CloudKit 対応改修（§5.3。リレーション optional 化 + デフォルト整備 + 移行検証）
  - Step 2: `ModelContainer` を CloudKit 構成へ、iCloud/CloudKit entitlements 追加
  - Step 3: 母語等のユーザー設定の扱い（当面は端末ローカル維持でも可）
  - Step 4: Settings に同期状態/iCloud サインイン状況の表示
  - バックエンドは共有 AI キャッシュのまま **改修なし**、共有シークレット継続。
- **Phase 2（必要になったら）**: シナリオ C（共有）や D（配布）。別スペックで判断。

想定が「不特定多数への配布（D）」なら、CloudKit では足りず、オプション 2 + §6 を含む
**プロダクト化の詳細設計** を別途起こす必要がある。

## §8. 確認したいこと（方針決定に必須）

1. **主目的はどれか**: (A) 自分の複数端末 / (B) 家族など少人数が各自独立 / (C) データ共有・協働 /
   (D) 不特定多数への配布。→ 以降すべてがこれで決まる。
2. **Apple 専用でよいか**: CloudKit 前提（iCloud 必須・Web/Android 不可）を許容できるか。
   将来 Web/Android を見据えるならオプション 2 寄りになる。
3. **AI 呼び出しコストの負担**: 現状は共有キーで作者が全額負担。ユーザーが増えたとき
   コスト上限/按分/BYO キーが要るか。
4. **データ共有（C）は要るか**: 「先生 - 生徒」的な共有・閲覧が要件に入るか、当面は各自独立で割り切れるか。

## §9. 影響範囲サマリ

- **iOS（CloudHat 路線の場合）**: `ESLLearningAssistantApp` の container 構成、Models のリレーション
  optional 化とデフォルト整備、entitlements（iCloud/CloudKit）、Settings UI、移行時のマイグレーション検証。
- **iOS（エクスポート/インポート）**: シリアライズ/デシリアライズと画面追加のみ。バックエンド影響なし。
- **バックエンド**: A/B + CloudKit 路線なら **改修なし**。オプション 2 や D では user テーブル・認証・
  同期 API・per-user メータリング等の大規模改修。
- **共有キャッシュ（words/illustrations/tts/quiz）**: どの路線でも個人データではないため原則そのまま活用可。
